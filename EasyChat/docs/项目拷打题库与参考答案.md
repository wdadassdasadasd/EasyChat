# 第二部分：参考答案

## 一、架构设计答案

### 1.1

项目采用 Electron 三层结构：主进程负责窗口、托盘、WebSocket、SQLite 和 IPC；preload 通过 `contextBridge` 暴露受控能力；renderer 使用 Vue 3 负责状态展示和交互。这样把系统能力、本地持久化与不可信页面环境隔离，UI 不直接操作数据库和 Node API。

### 1.2

WebSocket 和 SQLite 都属于应用级长期资源。放在主进程可以跨页面保持连接、统一控制生命周期、集中串行化数据库写入，也避免多个组件各建连接和直接并发写库。代价是主进程职责较重，因此需要按连接层、IPC 层、Model 层继续拆分。

### 1.3

会有这个风险。控制方法是让主进程只做能力编排：`wsClient` 管连接与接收队列，`ipc` 管跨进程协议，Model 管领域数据，`ADB` 管通用数据库原语；不能把所有业务都写进 `index.js`。进一步可引入 service 层和统一事件协议。

### 1.4

依赖方向应是 renderer -> preload API -> IPC handler -> service/model -> SQLite；实时事件反向由主进程通过白名单 channel 推送。renderer 不依赖 Model，Model 不依赖 Vue，数据库层也不感知 UI。这样每层可独立测试和替换。

### 1.5

桌面 IM 需要系统托盘、本地文件路径、大文件传输和稳定本地历史。SQLite 在复杂查询、事务、索引和大量数据持久化上比 localStorage 更合适，也比 IndexedDB 更容易做明确的数据模型与故障诊断。代价是引入 native 依赖和 Electron 打包复杂度。

### 1.6

多窗口会带来消息广播、当前活跃会话、已读状态和资源所有权问题。主进程仍应保持唯一 WebSocket 和 SQLite owner，再维护窗口订阅表，只把相关事件推给对应窗口；写操作要有 operationId，避免多个窗口重复提交。

### 1.7

服务端是跨设备事实源；SQLite 是本机可恢复事实源；Pinia 保存用户、联系人等跨页面会话状态；组件和 composable 保存当前聊天窗口的短生命周期状态。消息不全部塞进 Pinia，是为了避免全局 store 变成无限增长的大对象并扩大响应式开销。

### 1.8

我会保留三层结构、主进程持有 SQLite/WebSocket、先落库再推 UI、composable 拆分。重点重构 IPC 白名单、消息状态枚举、发送补偿任务、可恢复上传、连接状态机和结构化日志，并统一所有异步接口的错误语义。

## 二、消息链路答案

### 2.1

点击发送后先生成负数临时 ID 和 `status=2` 的 pending 消息，立即展示并通过 IPC 落 SQLite；随后 HTTP 创建服务端消息。成功后用真实 `messageId` 替换本地记录并改为成功，失败则标为 `status=0`，保留重试入口。对端再由 WebSocket 接收。

### 2.2

本地 pending 提供即时反馈，也让进程崩溃前有可恢复记录。负数 ID 不会与服务端正数 ID 冲突，并能参与列表 key 和数据库主键。更严谨的生产方案可以使用 `clientMessageId` UUID，把客户端幂等键和服务端序号分开。

### 2.3

HTTP 天然适合请求响应，能明确返回创建结果和服务端 ID；WebSocket 适合服务端主动推送和多端实时同步。也可以全部走 WebSocket，但需要自行实现 requestId、ACK、超时、重试和连接不可用时的降级，复杂度更高。

### 2.4

服务端已经成功，但本地可能仍保留 pending，重启后出现失败、重复或缺失。当前 UI 会标记 `localSyncFailed` 并等待后续同步，但生产级方案应落一条补偿任务，记录本地 ID、服务端 ID 和会话；重连或重新打开会话时按服务端数据修复。

### 2.5

renderer 用 `messageIdSet` 去重；替换 pending 时还会检查列表里是否已经存在真实服务端 ID，存在则合并并删除临时消息。更可靠的方式是服务端支持 `clientMessageId` 幂等，HTTP 响应和 WebSocket echo 都携带它。

### 2.6

串行队列可以保持提交顺序并减少状态竞态，但所有任务都串行会产生队头阻塞。文本创建请求可以按会话有序或有限并发；媒体上传应独立限流。当前项目把上传队列独立并限制为 3，是合理方向。

### 2.7

逐条处理会产生大量 SQLite 事务、IPC 事件和 Vue 更新，在群聊高峰或重连补推时放大开销。批量处理把多条消息合并为一次事务和一次 renderer 更新，同时能统一更新会话摘要、未读数和去重。

### 2.8

`50ms` 控制可感知延迟，`100` 条控制单批大小和内存增长，是延迟与吞吐的折中。不能凭感觉定死，应压测不同消息速率下的 P95 入库延迟、事务耗时、队列长度、renderer 长任务和内存，再动态调节。

### 2.9

数据库通过联合主键、批量插入前查询和批内 Set 去重；renderer 用 `messageIdSet` 防止重复 append。数据库去重保证持久化幂等，renderer 去重保证 UI 在 echo、重复 IPC 或历史分页交错时不闪重复，两者防护层次不同。

### 2.10

初始化消息需要同时恢复会话和最近历史，必须立即一致落库；状态 ACK 依赖已有消息，也要及时刷新队列后处理；普通消息可以短暂缓冲批量写入。分流是因为三类消息的时序和事务语义不同。

### 2.11

当前思路是处理 ACK 前先 flush 普通消息队列。但 ACK 仍可能早于 HTTP replace，生产级应把“未知消息 ACK”暂存在 orphan-ack 表或内存 Map，消息本体出现后再合并，并设置过期清理和服务端补拉。

### 2.12

每次会话加载绑定递增 `loadSeq`，回包时同时检查当前 `sessionId` 和序列号；不匹配就丢弃。它本质上是逻辑取消。若底层支持 AbortController，还可以同时取消网络请求，减少无效工作。

### 2.13

清空时记录 `clear_message_id` 或时间游标。历史查询和接收批量保存都过滤游标之前的消息，因此旧消息即使再次推送也不可见。这比物理删除更能表达“用户已清空到哪个同步位置”。

### 2.14

建议使用 `local_pending -> submitting -> server_created -> uploading -> sent -> delivered/read`，失败再区分 `send_failed/upload_failed/local_sync_failed/canceled`。当前 `0/1/2` 过于压缩，无法准确表达服务端已创建但文件未上传、本地替换失败等中间态。

## 三、文件上传 / 多媒体答案

### 3.1

统一链路负责 pending、创建服务端消息、状态持久化和重试；差异通过 `messageType`、文件元数据和上传 transport 注入。图片额外有本地预览和封面，视频有封面与大文件，普通文件主要展示名称、大小和下载状态。

### 3.2

先创建消息可以尽早获得稳定的服务端 `messageId`，后续分片都挂在这个 ID 下。上传是长耗时过程，失败只影响媒体内容，不阻塞整个会话；用户也能按消息维度取消和重试。

### 3.3

小文件单请求协议简单、额外握手少；大文件分片可重试、续传并降低单次失败成本。项目使用 8MB 阈值和 4MB 分片。真实选择要结合带宽、代理限制、服务端内存、请求开销和失败率压测。

### 3.4

init 创建上传会话并返回 `uploadId`、分片大小和已上传索引；chunk 负责传输单片并支持幂等；complete 校验片数、大小或 hash，合并文件并更新消息状态。三阶段把传输、恢复和最终提交分开。

### 3.5

客户端 init 时获取 `uploadedChunks`，用 Set 保存并跳过已存在的索引。服务端应以 `uploadId + chunkIndex` 唯一约束，并校验每片 hash。仅信任客户端索引不够，complete 时还要校验总大小和整体 hash。

### 3.6

并发 3 能利用带宽又不至于让多个大文件争抢 CPU、磁盘和连接。过大将导致网络拥塞、内存峰值和交互请求饥饿；过小则吞吐不足。更完善可根据网络类型、失败率和前后台状态动态调整。

### 3.7

先根据已完成分片累计 `uploadedBytes`，当前分片上传时加上 `event.loaded`，再除以总大小。显示上通常限制到 99%，只有 complete 成功才显示 100%，避免“传完但合并失败”造成假成功。

### 3.8

前端用 AbortController 终止当前请求、从队列移除任务并更新 UI；服务端 cancel 标记上传会话失效并异步清理临时分片。取消接口也要幂等，因为请求可能重复或在 complete 后才到达。

### 3.9

当前 `File` 和 `retryFile` 只存在内存，重启后无法重新获得浏览器 File 对象。要恢复，需要持久化原始文件路径、上传 ID、chunk size、已完成索引和文件指纹；启动时校验文件仍存在且未变化，再继续上传。

### 3.10

`URL.createObjectURL` 持有 Blob 引用，不释放会造成内存持续增长。消息被替换、会话切换、列表清空、组件卸载和发送完成不再需要本地预览时都要 revoke；同时避免释放仍被新列表复用的 URL。

## 四、SQLite 答案

### 4.1

SQLite 支持事务、索引、分页、全文搜索和大量结构化数据，适合桌面 IM 历史。localStorage 容量和查询能力不足；IndexedDB 可用但事务和调试模型不同；只依赖服务端会让离线历史、首屏速度和弱网体验变差。

### 4.2

`chat_message` 存消息明细；`chat_session_user` 存会话摘要、排序、置顶和未读数；`chat_session_clear` 存用户清空历史的同步边界。摘要表避免每次会话列表都扫描消息表，clear 表表达逻辑可见性。

### 4.3

同一台客户端会登录不同用户，服务端 `messageId` 未必需要跨租户唯一，因此联合用户 ID 隔离数据。它还能防止切换账号后误读另一用户消息。若服务端 ID 全局唯一，保留 user_id 仍有租户隔离价值。

### 4.4

`(user_id, session_id, message_id desc)` 支持按用户和会话取更早消息；会话表按 `(user_id, status, top_type desc, last_receive_time desc)` 支持过滤有效会话并按置顶和时间排序。索引顺序必须匹配常用 WHERE 和 ORDER BY。

### 4.5

WAL 让读操作不必阻塞写事务，适合 UI 查询与后台收消息并行；`busyTimeout` 在锁冲突时等待而非立即失败；`synchronous=NORMAL` 用少量极端断电耐久性换取更高写性能。聊天数据仍需服务端补偿作为最终保障。

### 4.6

WAL 允许一写多读，但 SQLite 同时仍只有一个 writer。应用层写队列能稳定写入顺序、减少 `SQLITE_BUSY`、让事务边界可控，并避免多个异步 Promise 争抢锁。WAL 和写队列解决的是不同层次问题。

### 4.7

若消息写入成功、摘要更新失败，会话列表最后一条和未读数就错误；反过来也可能摘要指向不存在的消息。同一事务保证消息明细、会话摘要和未读变化要么全部提交，要么全部回滚。

### 4.8

普通延迟事务可能到第一次写时才争抢锁；`BEGIN IMMEDIATE` 在事务开始就取得 reserved write lock，尽早暴露冲突。对于明确要写多张表的批处理，它能减少执行到中途才发现无法写入的情况。

### 4.9

把大数组按例如 500 个参数分批执行 `IN (...)` 查询，再合并结果。分片大小要为其他参数留余量。更大批量也可放入临时表后 join，但当前消息批次较小时分批最简单。

### 4.10

单纯删除无法阻止服务端补推旧消息再次出现。clear cursor 记录用户可见历史的下界，查询与入库都按它过滤。物理删除可以作为后续空间回收，但不能替代同步语义。

### 4.11

WAL checkpoint 把 WAL 中已提交页面同步回主数据库，并在合适条件下截断 WAL。长期不维护会让 WAL 增长、占用磁盘并拖慢恢复。项目按成功写入次数触发 PASSIVE checkpoint，还可在退出或空闲时做更积极维护。

### 4.12

前后通配的 `LIKE` 很难使用普通 B-tree 索引，历史变大后接近全表扫描。FTS5 建倒排索引，适合文本和文件名搜索，并支持相关性和分词。代价是需要维护索引一致性、迁移和中文 tokenizer 策略。

## 五、Electron / IPC 答案

### 5.1

主进程运行在 Node/Electron 环境，拥有系统与窗口权限；renderer 接近 Chromium 页面环境；preload 在页面加载前运行，是两者之间的受控桥梁。安全设计的目标是 renderer 即使出现 XSS，也拿不到任意 Node 能力。

### 5.2

`contextIsolation` 让 preload 与页面脚本拥有不同 JavaScript 上下文，页面不能篡改 preload 内部对象；`contextBridge` 只暴露明确 API。两者共同缩小攻击面，但前提是暴露的 API 本身也足够窄并校验参数。

### 5.3

完整 `ipcRenderer` 允许页面调用任意 channel，等于把主进程所有 IPC 能力暴露给潜在 XSS。当前可进一步改为 `chat.loadMessages`、`chat.sendLocalMessage` 等白名单函数，并在 preload 内固定 channel，不让页面自由传 channel 名。

### 5.4

`send/on` 是事件式、默认没有返回 Promise，适合广播和持续推送；`invoke/handle` 是一次请求一次响应，天然适合错误传递和 await。关键写操作和查询优先使用 invoke；实时消息、连接状态适合事件订阅。

### 5.5

renderer 可能被 XSS、插件或未来错误代码控制，IPC 是权限边界而不是可信函数调用。主进程必须校验 channel、字段类型、长度、用户身份、路径范围和操作权限，不能相信 renderer 传入的 SQL、文件路径或 userId。

### 5.6

向已销毁 `webContents` 发送会抛错或产生无效工作。项目发送前检查 sender 是否存在及 `isDestroyed()`，还用 runtime generation 丢弃旧连接的异步 flush。多窗口下还应移除已销毁窗口的订阅。

### 5.7

SQLite 是 native module，需要与 Electron 使用的 Node ABI 匹配，否则开发可用、打包后加载失败。通常通过 electron-builder 的依赖重建、正确 arch 打包和目标平台 CI 验证；不能直接拿一个平台生成的 node_modules 打所有平台。

### 5.8

主进程要对路径 `resolve` 后验证仍在允许目录，文件名取 basename，禁止 renderer 指定任意绝对目标；外链只允许 `https` 和可信域名，再用 `shell.openExternal`。还应关闭不必要的导航、新窗口和 Node integration。

## 六、Vue 3 答案

### 6.1

`Chat.vue` 只负责组合；会话、消息、发送、上传、滚动、虚拟列表各自拥有独立状态与生命周期。这样减少单文件复杂度，核心逻辑可在无 DOM 或少量 mock 下测试。拆分边界应按业务职责，不是机械地把函数搬文件。

### 6.2

基础值和会整体替换的数组适合 `ref`；多个相关字段对象适合 `reactive`；大 Map 或第三方对象不希望深度代理时用 `shallowRef`。项目高度缓存使用 Map，只通过版本号触发计算，避免深层响应式开销。

### 6.3

Map 的每次 set 不需要让 Vue 深度追踪，`shallowRef` 保持对象轻量；`heightVersion` 明确通知依赖重新计算。它把“数据容器”和“响应式失效信号”分开，适合高频测量更新。

### 6.4

这些值都可由消息数组、滚动位置、viewport 和高度缓存纯推导，使用 computed 能缓存结果并自动追踪依赖。副作用如滚动 DOM 不应放 computed，而应在事件或明确的调度函数中执行。

### 6.5

`nextTick` 等 Vue 把响应式变化 patch 到 DOM；`requestAnimationFrame` 再等浏览器进入下一帧，布局尺寸通常已经可读取。聊天列表 prepend、图片高度变化和滚动到底部都依赖真实布局，仅等待 Promise 返回不够。

### 6.6

稳定 key 让 Vue 正确复用消息组件和局部状态。数组下标在 prepend 后全部变化，会导致错误复用、重新渲染和高度缓存错位。服务端 `messageId` 或客户端临时 ID 都比 index 更合适。

### 6.7

用户身份、联系人通知等跨页面共享状态适合 Pinia；当前会话消息是局部、高频、可能很大的状态，留在聊天 composable 更容易清理和控制响应式范围。全局化不是复用的唯一方式，composable 同样能复用逻辑。

### 6.8

要移除 IPC listener、WebSocket/DOM listener、定时器、AbortController、Blob URL、未完成上传和虚拟列表缓存。否则会出现重复收消息、卸载后写状态、内存泄漏、重复 toast 和旧会话污染。

## 七、工程化答案

### 7.1

electron-vite 分别构建 main、preload、renderer，统一开发启动、HMR 和生产输出。开发时 renderer 走 Vite server 与代理；生产时加载打包后的静态资源，环境变量、资源路径和 native 依赖都要重新验证。

### 7.2

Vite 开发阶段利用浏览器原生 ESM，按需转换模块，不必每次先打完整 bundle；依赖预构建减少重复转换。renderer 可 HMR，main/preload 的变更通常需要对应进程重载，不能把三层 HMR 能力混为一谈。

### 7.3

Request 层统一 baseURL、token、loading、FormData、上传/下载进度、超时、取消、登录失效和错误提示。结构化错误把 timeout、network、HTTP status、业务 code、cancel 分开，调用方才能决定重试、退出登录或静默取消。

### 7.4

文件上传、带 AbortSignal、写操作和非幂等请求不能随便合并。即使 URL 和参数相同，两次“发送消息”也可能代表两个用户意图。去重只适合确定幂等的查询，并且 key 需要稳定序列化和及时清理。

### 7.5

单元测试覆盖状态机、去重、分片计算、虚拟列表和 Model；集成测试覆盖 IPC + SQLite、WebSocket batch + DB；E2E 覆盖登录、发送、断网重连、上传取消和重启恢复。核心故障逻辑以单元/集成为主，E2E 保留少量主路径。

### 7.6

必须测消息 pending/replace/fail、重复 echo、过期 loadSeq、队列失败恢复、事务回滚、分片续传和动态高度边界。不必对 Element Plus 内部 DOM、每个 class 名或纯视觉间距写脆弱断言，视觉问题更适合截图测试和人工 QA。

### 7.7

CI 至少运行 lint、Vitest、生产 build，并在 Windows 验证 Electron 打包和 SQLite native module。聊天链路修改应增加相关回归测试；可再做小规模压力测试，输出批量写吞吐、队列峰值和失败数。

### 7.8

优先级是：统一 UTF-8 编码和 IPC 白名单；补发送补偿与重启恢复；完善结构化日志和故障注入测试；再做 TypeScript、自动更新签名、跨平台打包和更完整 E2E。先解决数据正确性，再提升开发体验。

## 八、性能优化答案

### 8.1

主要瓶颈包括长消息列表 DOM、图片解码和布局、WebSocket 高频事件、SQLite 高频小事务、IPC 序列化、大文件上传与 Blob 内存。优化前先用消息量、长任务、事务耗时、内存和 FPS 定位，不能只凭感觉上虚拟列表。

### 8.2

每条消息有测量高度，前缀和 `acc[i]` 表示前 i 条总高度；根据 `scrollTop` 和 viewport 底部在前缀和中二分找 start/end，只渲染可视区加 overscan。顶部和底部 spacer 补齐未渲染区域，使滚动条总高度近似完整列表。

### 8.3

聊天历史会 prepend，数组下标会整体偏移，而 `messageId` 稳定。按 ID 缓存能复用旧消息高度，减少重新测量和滚动跳动。临时消息也应有稳定客户端 ID，不能长期依赖 index fallback。

### 8.4

图片加载后通过 ResizeObserver 或 load 事件重新测量该项高度，更新 Map 并从最早 dirty index 重算后续前缀和。如果用户靠近底部，还要维持贴底；若正在阅读历史，则不能强制跳到底部。

### 8.5

prepend 前记录虚拟总高度和 `scrollTop`；更新并完成布局后重新读取总高度，设置 `newScrollTop = oldScrollTop + newHeight - oldHeight`。这样新增历史占用的高度被补偿，原来视口中的消息保持原位置。

### 8.6

批量落库减少事务和 fsync 次数；批量 IPC 减少跨进程序列化与事件调度；批量 UI 更新减少 Vue patch 次数和布局抖动。三者分别优化存储、进程边界和渲染，缺一层都可能成为新瓶颈。

### 8.7

全量重算每次是 O(n)，大量图片逐个测量会接近 O(n²)。项目记录最早 `heightDirtyFrom`，只从该索引向后重算；还可把同一帧的多个测量合并，用 Fenwick Tree 将单点更新和前缀查询优化到 O(log n)。

### 8.8

对比优化前后的首屏时间、滚动 FPS、long task、DOM 节点数、内存、1000 条补推耗时、SQLite 事务次数、P95 入库到展示延迟。测试必须固定消息数据、机器和场景，并多轮运行，而不是只报一次最好结果。

## 九、稳定性答案

### 9.1

TCP 半开时本地 socket 可能仍显示 OPEN。只发送 ping 不检查 pong，无法证明对端可达；需要记录最后 pong，设置 deadline，超时后主动 terminate 并进入 reconnecting。项目已配置 pong timeout，回答时要说清状态转换。

### 9.2

用单一连接实例、`lockReconnect`、重连 timer 引用和 generation 管理生命周期；建新连接前清理旧 heartbeat、pong timer 和 handler。`onerror` 不再单独重连，由 `onclose` 统一触发，避免双重重连。

### 9.3

固定间隔会在服务恢复时产生客户端同时重连的惊群，也不适合长故障。可用 `min(cap, base * 2^attempt) + random jitter`，网络恢复或用户手动点击时立即重试，成功后重置 attempt。

### 9.4

若消息处理使用 Promise 链串行，一条永久 pending 会阻塞后续全部消息。需要给单任务加 timeout、catch 后继续链，并上报 `message_processing_timeout`；必要时把失败消息放隔离队列并触发会话 resync。

### 9.5

不能静默无限增长，也不能丢完不补。应设置上限和背压：优先暂停读取或降频，实在溢出时记录受影响 session 和最大已知 messageId，丢弃后触发增量历史同步。UI 要显示“正在恢复同步”，而非假装正常。

### 9.6

UI 已显示但本地未持久化，重启后消息消失，破坏用户信任。接收链路应先成功落库再推 UI；写失败时保持队列重试或进入 degraded/resync 状态。只有明确接受“临时显示”语义时才能先 UI 后落库。

### 9.7

登录失效应清理身份、关闭连接并跳登录；timeout/network 可提示重试并保留 pending；HTTP 500 表示服务暂不可用；cancel 通常静默或显示已取消。结构化 `kind` 让状态机采取不同动作，不能全部归为 null。

### 9.8

启动时扫描 `status=pending/uploading` 的记录：文本按超时规则转 failed 或使用 clientMessageId 查询服务端；媒体根据持久化路径和 uploadId 恢复分片，否则明确转为不可恢复失败。不能盲目重发，否则可能重复发送。

### 9.9

主进程事务内维护增量状态，重连 init 时以服务端快照校正；发现异常可按 session 重新计算摘要和未读，或直接拉服务端游标后的消息。所有修复都应幂等，并避免覆盖用户置顶等本地偏好。

### 9.10

记录结构化事件：连接状态、重连次数、lastPong、接收队列长度、drop 数、DB 读写错误、事务耗时、pending 年龄、上传失败类型和 resync 结果。日志带 session/message 的脱敏标识和 traceId，但不能打印 token 与私聊正文。

## 十、综合场景答案

### 10.1

文字先本地 pending 并快速提交，按会话保持顺序；视频创建消息后进入独立并发为 3 的上传队列，其余显示排队。网络失败分类处理，队列设上限，重连后用 clientMessageId 对账；退出前已持久化 pending 和上传元数据。

### 10.2

优先怀疑半开连接或 pong 处理失效。检查 lastPing/lastPong、socket readyState、心跳 timer、代理空闲超时和重连日志；模拟丢弃 pong 验证是否进入 stale -> reconnecting。若重连后一次性出现，说明服务端离线消息补推正常，客户端存活检测有问题。

### 10.3

主进程批量保存先按 messageId 去重，联合主键兜底；renderer 的 Set 再阻止重复 append，因此消息正文不会重复。会话未读累加也必须只基于实际新插入消息，否则虽然消息去重，未读数仍可能被加三次。

### 10.4

每次会话切换生成新的 loadSeq，历史回包校验 sessionId + loadSeq；实时消息根据自身 session 路由，非当前会话只 patch 会话摘要和未读，不写当前 messageList。滚动恢复状态也绑定同一加载批次，切换时清空。

### 10.5

需要持久化 uploadId、原文件路径、大小、mtime/hash、chunk size 和已完成分片。重启后校验文件未变化，调用 init 查询服务端已收分片，从缺失索引续传，最后 complete。若文件不存在或指纹变化，提示用户重新选择。

### 10.6

接收层设置有界队列并批量 100 条 flush；数据库事务去重并更新每个 session 的最新摘要；IPC 按批推送；当前会话使用虚拟列表，非当前会话只更新摘要。持续超载时触发背压和 resync，不让内存无限增长，并采集队列水位和处理延迟。

### 10.7

典型原因是 HTTP 已成功，但 `replacePendingMessage` 的 SQLite 事务失败，UI 当时使用内存态显示成功，重启后却读到旧 pending。应查看 localSyncFailed、DB 错误和服务端 messageId；修复方案是补偿任务或用 clientMessageId 从服务端对账，而不是直接重发。

### 10.8

端到端加密需要设备密钥、会话密钥、密文存储和密钥轮换；多端同步需要设备游标、消息版本、幂等操作日志和冲突策略；撤回需要不可变消息 ID、撤回事件、权限与时间校验。客户端数据模型应从“覆盖一行”演进为可重放的消息事件模型。

## 核心一：消息发送状态机

**对应问题：2.1、2.2、2.4、2.5、9.8**

代码位置：`src/renderer/src/views/chat/composables/useChatMessageSender.js`

```js
const sendChatMessage = async (
  { contactId, contactType, messageContent },
  retryMessage = null
) => {
  if (!messageContent) return

  // 首次发送创建负数临时 ID；重试则复用原消息，避免列表新增重复气泡。
  const localMessage =
    retryMessage ||
    createPendingMessage({
      contactId,
      contactType,
      messageType: 2,
      messageContent
    })

  if (retryMessage) {
    // 先把本地状态重新持久化为 sending，再发网络请求。
    await markMessageSending(localMessage)
  } else {
    // 先展示，给用户即时反馈。
    appendSentMessageIfMissing(localMessage)

    // 再落 SQLite。若本地都无法保存，就不继续网络发送，
    // 避免“当时看见成功，重启后消息消失”。
    await persistPendingMessage(localMessage)
  }

  const result = await proxy.Request({
    url: proxy.Api.sendMessage,
    params: {
      contactId,
      contactType,
      messageType: 2,
      messageContent
    },
    showLoading: false,
    returnError: true
  })

  if (!result || isRequestFailure(result)) {
    // 网络失败只改变这一条消息，保留内容和重试入口。
    await markMessageFailed(localMessage, getSendFailureMessage(result))
    return
  }

  try {
    // 使用服务端 messageId 原子替换本地负数 ID。
    await replaceLocalWithServerMessage(localMessage, result.data)
  } catch (error) {
    // 服务端已成功，不能提示用户直接重发，否则可能产生重复消息。
    markMessageLocalSyncFailed(localMessage, result.data, error)
  }
}
```

**口述答案：**

我的发送链路采用 local-first。消息先以负数临时 ID 和 pending 状态进入 UI 与 SQLite，再提交服务端；成功后用服务端 ID 替换，失败则保留原内容并允许重试。最关键的异常是“服务端成功但本地替换失败”，这时不能简单标记普通发送失败并让用户重发，而要标记为本地同步异常，后续通过服务端回显或补偿任务对账。

**面试官看重：**

- 是否理解即时反馈和持久化可靠性是两个问题。
- 是否能区分网络失败与本地同步失败。
- 是否考虑 HTTP 响应与 WebSocket echo 的竞态和幂等。

**高分补充：**

生产级方案会增加 `clientMessageId` UUID。HTTP 请求、WebSocket echo 和 SQLite 都携带该字段，由服务端按它保证幂等，比依赖负数 ID 更适合多端和重试场景。

## 核心三：WebSocket 接收批处理

**对应问题：2.7、2.8、2.9、8.6、9.5**

代码位置：`src/main/wsClient.js`

```js
const scheduleReceiveFlush = () => {
  if (receiveQueue.length >= RECEIVE_FLUSH_MAX) {
    // 数量达到阈值立即刷盘，限制队列增长和单条消息等待时间。
    flushReceiveQueue()
    return
  }

  if (!receiveFlushTimer) {
    // 第一条消息到达后只建立一个 timer，将短时间内消息合并。
    receiveFlushTimer = setTimeout(flushReceiveQueue, RECEIVE_FLUSH_DELAY)
  }
}

const flushReceiveQueue = async () => {
  clearReceiveFlushTimer()
  if (receiveFlushing) return // 防止 timer 和数量阈值并发触发两个 flush。
  receiveFlushing = true

  try {
    while (receiveQueue.length > 0) {
      // 单个事务有上限，避免一次处理过多导致主进程长时间占用。
      const messages = receiveQueue.slice(0, RECEIVE_FLUSH_MAX)

      try {
        // 内部顺序是：SQLite 保存成功 -> 批量 IPC 推送 renderer。
        await saveAndPublishMessageBatch(messages)
        receiveQueue.splice(0, messages.length)
      } catch (error) {
        messages.forEach((message) => {
          message.__receiveRetry = Number(message.__receiveRetry || 0) + 1
        })

        // 达到最大重试次数后移出主队列，防止一批坏消息永久队头阻塞。
        const failedMessages = messages.filter(
          (message) => message.__receiveRetry >= RECEIVE_SAVE_MAX_RETRY
        )

        if (failedMessages.length > 0) {
          const failedSet = new Set(failedMessages)
          receiveQueue = receiveQueue.filter((message) => !failedSet.has(message))

          // 通知上层按受影响会话重新同步，而不是静默丢弃。
          publishReceiveRecoveryNeeded({
            error,
            messages: failedMessages,
            stats: { kind: 'db_write_failed' }
          })
        }
        break
      }
    }
  } finally {
    receiveFlushing = false
    if (receiveQueue.length > 0) scheduleReceiveFlush()
  }
}
```

**口述答案：**

收到一条就落一次库会把消息洪峰放大成大量事务、IPC 和 Vue 更新，所以我使用时间阈值加数量阈值的双触发批处理。批次必须先成功落库再推 UI；失败时有限重试，超过阈值后触发 resync，避免坏消息永久堵塞后续队列。

**常见追问：为什么是 50ms 和 100 条？**

这两个值是延迟与吞吐的初始折中，不应声称是理论最优。应通过 P95 入库延迟、事务耗时、队列峰值、renderer 长任务和消息速率压测后调整。

## 核心四：半开连接和消息处理看门狗

**对应问题：9.1、9.2、9.3、9.4**

代码位置：`src/main/wsClient.js`

```js
const startHeartbeat = () => {
  clearHeartbeatTimer()

  pongHandler = () => {
    // pong 才能证明对端链路仍然可达。
    updateWsDiagnostics({ lastPongAt: Date.now(), lastError: '' })
    clearPongTimeoutTimer()
  }
  ws.on('pong', pongHandler)

  const sendHeartbeatPing = () => {
    // 上一个 ping 还没收到 pong，不重复堆积 heartbeat。
    if (awaitingPong || ws?.readyState !== WebSocket.OPEN) return

    ws.ping()
    updateWsDiagnostics({ lastPingAt: Date.now() })
    awaitingPong = true

    pongTimeoutTimer = setTimeout(() => {
      if (!awaitingPong) return

      // socket 显示 OPEN 不代表链路真实可用，将其标记为 stale。
      publishWsStatus({
        status: 'stale',
        error: 'WebSocket heartbeat timed out'
      })

      clearHeartbeatTimer()
      closeCurrentSocket()
      reconnect()
    }, HEARTBEAT_PONG_TIMEOUT)
  }

  sendHeartbeatPing()
  heartbeatTimer = setInterval(sendHeartbeatPing, HEARTBEAT_INTERVAL)
}

const enqueueMessageProcessing = (message) => {
  messageProcessingQueue = messageProcessingQueue
    .catch(() => {
      // 前一个任务失败不能让整条 Promise 链保持 rejected。
    })
    .then(() =>
      // 单条处理超过限制就主动失败，避免后续所有消息永久等待。
      runWithTimeout(
        () => handleWsMessage(message),
        WS_MESSAGE_PROCESS_TIMEOUT,
        'WebSocket message processing timed out'
      )
    )
    .catch((error) => {
      publishReceiveRecoveryNeeded({
        error,
        messages: normalizeWsMessages(message).filter(isValidWsMessage),
        stats: { kind: 'message_processing_timeout' }
      })
    })

  return messageProcessingQueue
}
```

**口述答案：**

心跳不能只看 `readyState` 或只发送 ping，因为 TCP 半开时本地仍可能认为连接正常。必须等待 pong，并在 deadline 内未收到时主动关闭、进入 stale 和 reconnecting。消息处理队列也需要看门狗，否则一个永不 resolve 的 Promise 会阻塞所有后续消息。

**高频追问：重连为什么要加 jitter？**

大量客户端同时断线时，固定 5 秒重连会在服务恢复瞬间形成惊群。生产级可采用 `min(cap, base * 2^attempt) + random jitter`，连接成功后重置 attempt。

```js
  // 已在事务内部时直接执行，避免事务中的写操作再次排队导致自锁。

        await runStrictNow('commit')
```


**口述答案：**

大文件先 init 获取上传会话和已完成分片，然后逐片上传，最后 complete 合并。重试时跳过服务端已经确认的分片，因此失败成本只是一片。当前实现能在本次运行中续传；若要支持应用重启恢复，还要持久化原始路径、uploadId、分片大小、文件 hash 和修改时间。

**面试官可能继续问：为什么分片内串行，文件间并发？**

单文件串行让进度、服务端合并和失败恢复更简单；多个文件最多 3 个并发用于提高整体吞吐。后续可以在带宽充足时允许单文件有限并发，但要增加分片调度和乱序完成处理。

## 核心七：动态高度虚拟列表

**对应问题：8.2、8.3、8.4、8.5、8.7**

代码位置：`src/renderer/src/views/chat/composables/useVirtualMessageList.js`

```js
const heightMap = shallowRef(new Map())
const heightVersion = ref(0)
let heightDirtyFrom = -1
let prevAcc = []

const setMessageHeight = (message, index, height) => {
  // messageId 在 prepend 后不变，比数组下标更适合作为高度缓存 key。
  const key = String(message.messageId)
  if (heightMap.value.get(key) === height) return

  heightMap.value.set(key, height)
  heightDirtyFrom =
    heightDirtyFrom < 0 ? index : Math.min(heightDirtyFrom, index)

  // Map 不做深度响应式，使用轻量版本号通知 computed 失效。
  heightVersion.value += 1
}

const accumulatedHeights = computed(() => {
  void heightVersion.value
  const count = messageList.value.length

  if (prevAcc.length !== count + 1) {
    // 列表长度变化时重建前缀和：
    // acc[i] 表示前 i 条消息的总高度。
    prevAcc = new Array(count + 1)
    prevAcc[0] = 0
    for (let i = 0; i < count; i++) {
      prevAcc[i + 1] = prevAcc[i] + getHeight(i)
    }
  } else {
    // 图片加载只改变局部高度，从最早脏索引向后重算。
    for (let i = heightDirtyFrom; i < count; i++) {
      prevAcc[i + 1] = prevAcc[i] + getHeight(i)
    }
  }

  heightDirtyFrom = -1
  return prevAcc
})

const upperBound = (acc, target, left, right) => {
  while (left < right) {
    const mid = (left + right) >>> 1
    if (acc[mid] <= target) left = mid + 1
    else right = mid
  }
  return left
}

const startIndex = computed(() => {
  // 在前缀和中二分定位 scrollTop，再额外渲染 overscan。
  const index = upperBound(
    accumulatedHeights.value,
    scrollTop.value,
    0,
    messageList.value.length
  )
  return Math.max(0, index - overscan)
})
```

**口述答案：**

聊天消息高度不固定，不能使用固定行高虚拟列表。我按 `messageId` 缓存实测高度，用前缀和表示任意消息之前的累计高度，再对 `scrollTop` 二分查找可视起点，只渲染可视区加 overscan。图片加载导致高度变化时，从最早脏索引增量重算，避免每次测量都全量 O(n)。

**复杂度：**

- 定位可视区：O(log n)。
- 切片渲染：O(k)，k 是可视消息数加 overscan。
- 当前高度更新：最坏 O(n)，但比每条测量都全量重算更好；进一步可用 Fenwick Tree 优化到 O(log n)。
