## 三、接口契约验证

### 响应格式

所有端点统一返回 JSON：
```json
{
  "status": "success" | "error",
  "code": 200 | 600 | 901 | 404,
  "info": "描述信息",
  "data": { ... } | null
}
```
✅ 前端 `Request.js` 的响应拦截器（检查 `code == 200` / `code == 901`）与此格式完全对齐。

### 认证流程

| 步骤 | 行为 | 验证 |
|------|------|------|
| Token 传输 | `token` header（小写） | ✅ Request.js 第 178 行设置 |
| Token 无效 | 返回 `code: 901` | ✅ 39 个端点一致 |
| 901 处理 | 前端清除 userInfo + navigate /login | ✅ Request.js resetLoginState() |
| 登录/注册 | 跳过 token 携带 | ✅ Request.js 前缀 `/account/` 判断 |

### 请求格式

| 前端行为 | 后端期望 | 匹配 |
|----------|----------|------|
| 默认 `Content-Type: application/x-www-form-urlencoded` | ✅ | 所有 POST 正常 |
| 文件上传切换 `multipart/form-data` | 待验证 | 需要实际文件上传测试 |
| JSON Content-Type | ⚠️ 600 | 后端不接收 JSON body |
| GET 请求 | ✅ 部分端点支持 | checkCode 可 GET，其余需 POST |
| `/api` 前缀 | ✅ 必需 | 不带前缀返回 Tomcat 404 HTML |

## 五、发现的 BUG

### 🟡 Bug: 2 个更新管理端点未部署

**文件**: `src/renderer/src/utils/Api.js`

| 前端调用 | 后端状态 |
|----------|----------|
| `Api.loadUpdateDataList` → `/update/loadUpdateList` | 404 |
| `Api.delUpdate` → `/update/delUpdate` | 404 |

**影响**: 更新管理功能（管理员上传新版本、删除旧版本）的前端按钮点击后会得到 404 错误。如果前端未处理该错误，用户会看到"请求地址不存在"的提示。

**建议**: 
- 如果后端计划支持：在 Spring Boot 中添加对应 Controller
- 如果后端不支持：在 `Api.js` 中移除这两个端点，或在前端隐藏相关 UI

## 七、建议

1. **修复 404 端点**: 确认 `update/loadUpdateList` 和 `update/delUpdate` 是否需要，避免前端调用报错
2. **登录流程端到端测试**: 需要人工输入图片验证码完成完整登录 → WebSocket 连接 → 收发消息流程
